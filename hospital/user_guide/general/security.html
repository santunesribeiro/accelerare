Segurança¶

Esta página descreve algumas "melhores práticas" em relação à segurança da web e detalha os recursos internos de segurança do CodeIgniter.

Nota

Se você veio aqui procurando um contato de segurança, consulte o nosso Guia de Contribuição <../contributing/index>.

Segurança de URI¶

O CodeIgniter é bastante restritivo em relação aos caracteres que permite nas suas cadeias URI para ajudar a minimizar a possibilidade de dados mal-intencionados serem passados ​​para sua aplicação. Os URIs só podem conter o seguinte:

  • Texto alfanumérico (caracteres em latin somente)
  • Til ~
  • Sinal de porcentagem: %
  • Ponto .
  • Dois Pontos :
  • Sublinhado _
  • Traço: -
  • Espaço

Register_globals¶

Durante a inicialização do sistema, todas as variáveis ​​globais que existem em $_GET, $_POST, $_REQUEST e $_COOKIE são desatualizadas.

A rotina desatualiza é efetivamente a mesma que register_globals = off.

display_errors¶

Em ambientes de produção, normalmente é desejável "desativar" o relatório de erros do PHP configurando o sinalizador display_errors interno para um valor de 0. Isso desabilita que os erros nativos do PHP sejam processados ​​como saída, o que potencialmente pode conter informações confidenciais.

Configurando a constante ENVIROMENT do CodeIgniter em index.php para um valor de 'produção' desligará esses erros. No modo de desenvolvimento, recomenda-se que seja utilizado um valor de "desenvolvimento". Mais informações sobre a diferenciação entre ambientes podem ser encontradas na página Ambientes de Manipulação.

magic_quotes_runtime¶

A diretiva magic_quotes_runtime é desativada durante a inicialização do sistema, de modo que você não precise remover as barras ao recuperar dados do banco de dados.

Boas Práticas¶

Antes de aceitar qualquer dado em seu aplicativo, seja ele dados POST de um envio de formulários, dados COOKIE, dados URI, dados XML-RPC ou mesmo dados da matriz SERVER, você é encorajado a praticar esta abordagem de três passos:

  1. Valide os dados para garantir que ele esteja em conformidade com o tipo correto, comprimento, tamanho, etc.
  2. Filtre os dados como se estivesse contaminado.
  3. Escape dos dados antes de enviá-lo para o seu banco de dados ou enviá-lo para um navegador.

O CodeIgniter fornece as seguintes funções e dicas para ajudá-lo neste processo:

Filtrando XSS¶

O CodeIgniter vem com um filtro Cross Site Scripting. Este filtro procura técnicas comumente usadas para incorporar JavaScript malicioso em seus dados ou outros tipos de código que tentam seqüestrar cookies ou fazer outras coisas maliciosas. O Filtro XSS é descrito aqui.

Nota

A filtragem XSS só deve ser realizada na saída. Os dados de entrada de filtragem podem modificar os dados de maneiras indesejáveis, incluindo descartar caracteres especiais de senhas, o que reduz a segurança em vez de melhorá-la.

Proteção CSRF¶

CSRF significa Cross-Site Request Forgery, que é o processo de um atacante enganando sua vítima ao enviar um pedido inadvertidamente.

O CodeIgniter fornece proteção CSRF fora da caixa, que será ativado automaticamente para cada solicitação HTTP não GET, mas também precisa de criar seus formulários de envio de uma determinada maneira. Isso é explicado na documentação da Biblioteca de Segurança.

Manipulação de senha¶

É fundamental que você manipule senhas em seu aplicativo corretamente.

Infelizmente, muitos desenvolvedores não sabem como fazer isso, e a web está cheia de conselhos desatualizados ou de outra forma incorretos, o que não ajuda.

Gostaríamos de lhe dar uma lista de "fazer e não fazer" para ajudá-lo com isso. Veja a lista abaixo.

  • NÃO armazene senhas em formato de texto simples.

    Sempre faça hash das suas senhas.

  • NÃO use Base64 ou codificação similar para armazenar senhas.

    Isso é tão bom quanto armazená-los em texto simples. Mesmo. Faça hashing, não codifique.

    A codificação e a criptografia também são processos bidirecionais. As senhas são segredos que só devem ser conhecidos do seu proprietário e, portanto, devem funcionar apenas em uma direção. Hashing faz isso - não há nenhum-hashing ou de-hashing, mas há decodificação e descriptografia.

  • NÃO USE algoritmos de hash fracos ou quebrados como MD5 ou SHA1.

    Esses algoritmos são antigos, provados ser defeituosos e não projetados para hash de senha em primeiro lugar.

    Além disso, NÃO CRIE seus próprios algoritmos.

    Use apenas algoritmos de hashing de senha forte como o BCrypt, que é usado nas próprias funções de Hashing de Senha do PHP.

    Por favor, use-os, mesmo se você não estiver executando o PHP 5.5, o CodeIgniter os fornece para você, desde que você esteja executando pelo menos a versão PHP 5.3.7 (e se você não atender a esse requisito - por favor, atualize).

    Se você é uma das pessoas realmente desafortunadas que não podem sequer atualizar para uma versão mais recente do PHP, use hash_pbkdf() <http://php.net/hash_pbkdf2>, que também fornecemos na nossa camada de compatibilidade.

  • NUNCA exiba ou envie uma senha em formato de texto simples!

    Mesmo para o proprietário da senha, se você precisar de um recurso "Senha esquecida", apenas gere aleatoriamente uma senha nova, única (também é importante) e envie isso.

  • NÃO coloque limites desnecessários nas senhas de seus usuários.

    Se você estiver usando um algoritmo de hash diferente de BCrypt (que tem um limite de 72 caracteres), você deve definir um limite relativamente alto de comprimentos de senha para mitigar ataques DoS - digamos, 1024 caracteres.

    Além disso, não há nenhum ponto em forçar uma regra de que uma senha só pode ser até um número de caracteres ou que não pode conter um determinado conjunto de caracteres especiais.

    Isso não só reduz a segurança em vez de melhorar, mas não há literalmente nenhuma razão para fazê-lo. Não há limitações técnicas e não existem restrições de armazenamento (práticas) quando você as hashed, nenhuma!

Validando dados de entrada¶

O CodeIgniter tem uma Biblioteca de Validação de Formulário que ajuda você a validar, filtrar e preparar seus dados.

Mesmo que isso não funcione para o seu caso de uso no entanto, certifique-se de sempre validar e desinfetar todos os dados de entrada. Por exemplo, se você espera uma string numérica para uma variável de entrada, pode verificar isso com is_numeric() ou ctype_digit(). Sempre tente restringir suas verificações para um determinado padrão.

Tenha em mente que isso inclui não só as variáveis ​​$_POST e $_GET, mas também os cookies, a seqüência do agente do usuário e, basicamente, todos os dados que não são criados diretamente pelo seu próprio código.

Escape de Todos os Dados Antes da Inserção no Banco de Dados¶

Nunca insira informações em sua base de dados sem escapar. Consulte a seção que discute consultas do banco de dados para obter mais informações.

Esconda seus arquivos¶

Outra boa prática de segurança é deixar seu index.php e "ativos" (por exemplo .js, css e arquivos de imagem) no diretório webroot do seu servidor (mais comumente chamado de "htdocs /"). Estes são os únicos arquivos que você precisa estar acessível a partir da web.

Permitir que seus visitantes vejam algo mais potencialmente lhes permitirá acessar dados confidenciais, executar scripts, etc.

Se você não tem permissão para fazer isso, você pode tentar usar um arquivo .htaccess para restringir o acesso a esses recursos.

O CodeIgniter terá um arquivo index.html em todos os seus diretórios na tentativa de ocultar alguns desses dados, mas tenha em mente que isso não é suficiente para evitar um invasor sério.
