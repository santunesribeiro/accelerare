Roteamento URI¶

Normalmente, existe uma relação de um para um entre uma string de URL e sua classe/método do controlador correspondente. Os segmentos em um URI normalmente seguem esse padrão:

exemplo.com/classe/funcao/id/

Em alguns casos, no entanto, você pode querer remapear esse relacionamento para que uma classe/método diferente possa ser chamado em vez da correspondente ao URL.

Por exemplo, digamos que deseja que seus URLs tenham esse protótipo:

exemplo.com/produto/1/
exemplo.com/produto/2/
exemplo.com/produto/3/
exemplo.com/produto/4/

Normalmente, o segundo segmento do URL é reservado para o nome do método, mas no exemplo acima, ele possui uma ID de produto. Para superar isso o CodeIgniter permite que você remapeie o manipulador URI.

Configurando suas próprias regras de roteamento¶

As regras de roteamento são definidas no arquivo application/config/routes.php. Nela você verá uma matriz chamada $route que permite que você especifique seus próprios critérios de roteamento. As rotas podem ser especificadas usando curingas ou Expressões Regulares

Curingas¶

Uma rota curinga típica pode parecer algo assim:

$route['product/:num'] = 'catalog/product_lookup';

Em uma rota, a chave da matriz contém o URI a ser combinado, enquanto o valor da matriz contém o destino para o qual ele deve ser encaminhado. No exemplo acima, se a palavra literal “product” É encontrado no primeiro segmento do URL, e um número é encontrado no segundo segmento, a classe “catalog” e o método “product_lookup” são uados em vez disso.

Você pode combinar valores literais ou pode usar dois tipos curinga:

(:num) Combinará um segmento contendo apenas números. (:any) Irá combinar um segmento contendo qualquer caractere (exceto para ‘/’, qual é o delimitador de segmento).

Nota

Os curingas são realmente alias para expressões regulares, com :any sendo traduzido para [^/]+ e :num para [0-9]+, respectivamente.

Nota

As rotas serão executadas na ordem em que são definidas. As rotas mais elevadas prevalecerão sempre sobre os mais baixos.

Note

As regras da rota não são filtros! Definir uma regra de e.g. ‘foo/bar/(:num)’ não impedirá o controlador Foo e o método bar Para ser chamado com um valor não numérico se essa for uma rota válida.

Exemplos¶

Aqui estão alguns exemplos de roteamento:

$route['journals'] = 'blogs';

Uma URL contendo a palavra “journals” no primeiro segmento será remapeado para a classe “blogs”.

$route['blog/joe'] = 'blogs/users/34';

Um URL contendo os segmentos blog/joe será remapeado para a classe “blogs” e o método “users”. O ID será configurado para “34”.

$route['product/(:any)'] = 'catalog/product_lookup';

A URL com “product” como o primeiro segmento, e qualquer coisa no segundo será remapeada para a classe “catalog” e o método “product_lookup”.

$route['product/(:num)'] = 'catalog/product_lookup_by_id/$1';

A URL com “product” como o primeiro segmento, e um número no segundo será remapeado para a classe “catalog” e o método “product_lookup_by_id” Passando a combinação como uma variável para o método.

Importante

Não use barras diagonais.

Expressão Regular¶

Se você preferir, você pode usar expressões regulares para definir suas regras de roteamento. Qualquer expressão regular válida é permitida, assim como as referências de volta.

Nota

Se você usa referências de retorno, você deve usar a sintaxe do dólar em vez da sintaxe da barra invertida dupla.

Uma rota RegEx típica pode parecer algo assim:

$route['products/([a-z]+)/(\d+)'] = '$1/id_$2';

No exemplo acima, um URI asemelha-se com products/shirts/123 Em vez disso, chamaria a classe do controlador “shirts” e o método “id_123”.

Com expressões regulares, você também pode pegar vários segmentos ao mesmo tempo. Por exemplo, se um usuário acessa uma área protegida por senha de seu aplicativo da web e você deseja redirecioná-los de volta para a mesma página depois de fazer logon, você pode achar esse exemplo útil:

$route['login/(.+)'] = 'auth/login/$1';

Nota

No exemplo acima, se $1 contém uma barra, ele ainda será dividido em vários parâmetros quando passado para Auth::login().

Para aqueles que não conhecem expressões regulares e desejam aprender mais sobre elas, regular-expressions.info pode ser um bom começo.

Nota

Você também pode misturar e combinar curingas com expressões regulares.

Callbacks¶

Se você estiver utilizando PHP >= 5.3 você pode usar callbacks no lugar das regras normais de roteamento para processar as referências de retorno. Exemplo:

$route['products/([a-zA-Z]+)/edit/(\d+)'] = function ($product_type, $id)
{
        return 'catalog/product_edit/' . strtolower($product_type) . '/' . $id;
};

Usando verbos em rotas HTTP¶

É possível usar verbos HTTP (método de solicitação) para definir suas regras de roteamento. Isso é particularmente útil ao criar aplicativos RESTful. Você pode usar verbos HTTP padrão (GET, PUT, POST, DELETE, PATCH) ou um personalizado tal (por exemplo, PURGE). As regras do verbo HTTP não são sensíveis a maiúsculas e minúsculas. Tudo o que você precisa fazer é adicionar o verbo como uma chave de matriz para sua rota. Exemplo:

$route['products']['put'] = 'product/insert';

No exemplo acima, uma solicitação PUT para URI “products” que chamaria a classe com o método Product::insert().

$route['products/(:num)']['DELETE'] = 'product/delete/$1';

Um pedido DELETE para URL com “products” Como primeiro o segmento e um número no segundo serão mapeados para a classe e o método Product::delete(), passando um valor numérico para o primeiro parâmetro.

Usar verbos HTTP é claro, opcional.

Rotas Reservadas¶

Existem três rotas reservadas::

$route['default_controller'] = 'welcome';

Esta rota aponta para a ação que deve ser executada se o URI não contém dados, o que será o caso quando as pessoas carregarem seu URL de raiz. A configuração aceita um valor para o controller/method e index() seria o método padrão se você não especificar um. No exemplo acima, é Welcome::index() que seria chamado.

Nota

Você NÃO pode usar um diretório como parte desta configuração!

Você é encorajado a ter sempre uma rota padrão, caso contrário, uma página 404 aparecerá por padrão.

$route['404_override'] = '';

Esta rota indica qual classe de controlador deve ser carregada se o controlador solicitado não for encontrado. Ele substituirá a página de erro 404 padrão. As mesmas regras por diretório que com ‘default_controller’ aplica-se aqui também.

Isso não afetará a função show_404(), que continuará a carregar o arquivo padrão error_404.php em application/views/errors/error_404.php.

$route['translate_uri_dashes'] = FALSE;

Como é evidente pelo valor booleano, esta não é exatamente uma rota. Esta opção permite que você substitua automaticamente os traços (‘-‘) com sublinhados nos segmentos URI do controlador e do método, salvando assim as entradas de rota adicionais se você precisar fazer isso. Isso é necessário, porque o traço não é um caractere de nome de classe ou método válido e causaria um erro fatal se você tentar usá-lo.

Importante

As rotas reservadas devem vir antes de qualquer curinga ou rotas de expressões regulares.
