Ganchos - Estendendo o Núcleo do Framework¶

O recurso de ganchos do CodeIgniter fornece um meio para explorar e modificar o funcionamento interno da estrutura sem hackear os arquivos principais. Quando CodeIgniter é executado, segue um processo de execução específico, descrito na página de Fluxo de aplicação. Pode haver casos, no entanto, onde você gostaria de fazer alguma ação ocorrer em uma etapa específica no processo de execução. Por exemplo, você pode querer executar um script antes de seus controladores serem carregados ou logo depois, ou você pode querer desencadear um de seus próprios scripts em algum outro local.

Habilitando Ganchos¶

O recurso de ganchos pode ser ativado/desativado globalmente, configurando o seguinte item no arquivo application/config/config.php:

$config['enable_hooks'] = TRUE;

Definindo um Gancho¶

Os ganchos são definidos no arquivo application/config/hooks.php. Cada gancho é especificado como uma matriz com este protótipo:

$hook['pre_controller'] = array(
        'class'    => 'MyClass',
        'function' => 'Myfunction',
        'filename' => 'Myclass.php',
        'filepath' => 'hooks',
        'params'   => array('beer', 'wine', 'snacks')
);

Nota:

O índice da matriz correlaciona-se ao nome do ponto de gancho particular que deseja usar. No exemplo acima, o ponto de gancho é pré-controlador. Uma lista de pontos de gancho é encontrada abaixo. Os seguintes itens devem ser definidos em sua matriz de gancho associativo:

  • class O nome da classe que você deseja invocar. Se você preferir usar uma função processual em vez de uma classe, deixe este item em branco.
  • function O nome da função (ou do método) que você deseja chamar.
  • filename O nome do arquivo que contém sua classe/função.
  • filepath O nome do diretório que contém seu script. Nota: Seu script deve estar localizado DENTRO de seu diretório application/directory, de modo que o caminho do arquivo seja relativo a esse diretório. Por exemplo, se seu script estiver localizado em application/hooks/, você simplesmente usará 'hooks' como seu caminho de arquivo. Se o seu script estiver localizado em application/hooks/utilities/ você usará 'hooks/utilities' como caminho do arquivo. Sem barra diagonal.
  • params Todos os parâmetros que você deseja transmitir ao seu script. Este item é opcional.

Se você estiver executando o PHP 5.3+, você também pode usar funções lambda/anônimas (ou fechadas) como ganchos, com uma sintaxe mais simples:

$hook['post_controller'] = function()
{
        /* alguma coisa aqui */
};

Várias Chamadas para o Mesmo Gancho¶

Se quiser usar o mesmo ponto de gancho com mais de um script, basta fazer a sua declaração de matriz multidimensional, assim:

$hook['pre_controller'][] = array(
        'class'    => 'MyClass',
        'function' => 'MyMethod',
        'filename' => 'Myclass.php',
        'filepath' => 'hooks',
        'params'   => array('beer', 'wine', 'snacks')
);

$hook['pre_controller'][] = array(
        'class'    => 'MyOtherClass',
        'function' => 'MyOtherMethod',
        'filename' => 'Myotherclass.php',
        'filepath' => 'hooks',
        'params'   => array('red', 'yellow', 'blue')
);

Observe os colchetes após cada índice de matriz:

$hook['pre_controller'][]

Isso permite que você tenha o mesmo ponto de gancho com vários scripts. A ordem que você definiu sua matriz será a ordem de execução.

Pontos de Gancho¶

O seguinte é uma lista de pontos de gancho disponíveis.

  • pre_system Chamado muito cedo durante a execução do sistema. Somente a classe benchmark e hooks foi carregada neste momento. Nenhum roteamento ou outros processos aconteceram.
  • pre_controller Chamado imediatamente antes de qualquer um dos seus controladores serem chamados. Todas as classes base, roteamento e verificações de segurança foram feitas.
  • post_controller_constructor Chamado imediatamente após o seu controlador ser instanciado, mas antes de qualquer método de chamadas acontecer..
  • post_controller Called immediately after your controller is fully executed.
  • display_override Substitui o método _display() usado para enviar a página finalizada para o navegador da Web no final da execução do sistema. Isso permite que você use sua própria metodologia de exibição. Observe que você precisará fazer referência ao super objeto do CI com $this->CI =& get_instance() e então os dados finalizados estarão disponíveis chamando $this->CI->output->get_output().
  • cache_override Permite-lhe chamar o seu próprio método em vez do método _display_cache() da Biblioteca de saída. Isso permite que você use seu próprio mecanismo de exibição de cache.
  • post_system Chamado depois que a página renderizada final é enviada para o navegador, no final da execução do sistema depois que os dados finalizados são enviados para o navegador.
